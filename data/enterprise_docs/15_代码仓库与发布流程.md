## 一、代码仓库管理

### 1.1 仓库平台

公司使用**GitLab私有部署**作为统一代码托管平台（地址：https://gitlab.xingchen-tech.com），所有项目代码必须提交至公司GitLab仓库，禁止使用外部代码托管平台（如GitHub、Gitee等）存储公司项目代码。

### 1.2 分支策略

采用Git Flow分支管理模型：

| 分支类型 | 命名规范 | 用途 | 生命周期 |
|----------|----------|------|----------|
| main | main | 生产环境代码，始终保持可发布状态 | 永久 |
| develop | develop | 开发集成分支，包含最新开发完成的功能 | 永久 |
| feature/* | feature/JIRA编号-简述 | 新功能开发 | 合并后删除 |
| hotfix/* | hotfix/JIRA编号-简述 | 线上紧急修复 | 合并后删除 |
| release/* | release/v版本号 | 发布准备分支 | 发布完成后删除 |

**分支保护规则：**
- main和develop分支设为保护分支，禁止直接push
- 所有变更必须通过Merge Request（MR）合并
- MR标题必须关联JIRA编号（格式：[PROJ-123] 功能描述）

## 二、代码审查

### 2.1 审查要求

| 变更类型 | 最低审查人数 | 额外要求 |
|----------|-------------|----------|
| 普通功能变更 | 1位reviewer批准 | — |
| 核心模块变更 | 2位reviewer批准 | 需包含1位高级工程师 |
| 安全相关变更 | 2位reviewer + 安全团队 | 需安全工程师审查并签字 |
| 数据库Schema变更 | 2位reviewer + DBA | 需DBA评估SQL性能影响 |

安全相关变更包括但不限于：认证授权逻辑修改、加密算法变更、外部输入处理、权限配置调整。具体判断标准参考《信息安全管理制度》（文档编号：SEC-2026-003）中的安全编码规范章节。

### 2.2 审查标准

- 代码风格符合项目ESLint/Pylint规则
- 单元测试覆盖率不低于变更行数的80%
- 无明显性能问题（N+1查询、大循环等）
- 无硬编码配置（数据库密码、API Key等）
- 日志输出规范，不包含敏感信息

## 三、CI/CD流水线

### 3.1 流水线架构

使用GitLab CI作为持续集成/持续部署平台，流水线阶段如下：

| 阶段 | 触发条件 | 执行内容 | 时限要求 |
|------|----------|----------|----------|
| Lint | 所有MR | 代码风格检查、静态分析 | 5分钟 |
| Unit Test | 所有MR | 单元测试执行 + 覆盖率报告 | 15分钟 |
| Integration Test | develop合并 | 集成测试、API契约测试 | 30分钟 |
| Security Scan | develop合并 | SAST静态安全扫描 + 依赖漏洞扫描 | 10分钟 |
| Build | 发布分支 | 构建Docker镜像、推送至镜像仓库 | 10分钟 |
| Deploy Staging | 发布分支 | 部署至Staging环境 | 15分钟 |
| Deploy Prod | 手动触发 | 灰度部署至生产环境 | 按发布流程 |

### 3.2 质量门禁

Merge Request合并必须满足以下条件：

- [x] 所有CI阶段通过（无失败用例）
- [x] 单元测试覆盖率 >= 70%（新增代码覆盖率 >= 80%）
- [x] 无Critical/High级别安全漏洞
- [x] 至少1位reviewer批准
- [x] 无合并冲突

## 四、发布流程

### 4.1 发布窗口

| 类型 | 时间 | 审批要求 |
|------|------|----------|
| 常规发布 | 每周二、周四 14:00-17:00 | 技术负责人审批 |
| 紧急修复 | 任何时间 | 技术总监审批 + 事后补发报告 |
| 重大版本发布 | 每月最后一个周四 | CTO审批 |

### 4.2 发布前检查清单

| 检查项 | 负责人 | 状态 |
|--------|--------|------|
| Staging环境验证通过 | QA工程师 | □ |
| 数据库变更脚本准备就绪 | DBA | □ |
| 回滚方案已准备 | 发布工程师 | □ |
| 监控告警配置已更新 | 运维工程师 | □ |
| 发布通知已发送 | 项目经理 | □ |
| 客户影响评估完成 | 产品经理 | □ |

### 4.3 灰度发布策略

采用渐进式灰度发布，确保线上稳定性：

| 阶段 | 流量比例 | 观察时长 | 通过标准 |
|------|----------|----------|----------|
| 第1阶段 | 5% | 30分钟 | 错误率 < 0.1%，P99延迟 < 500ms |
| 第2阶段 | 20% | 30分钟 | 错误率 < 0.1%，P99延迟 < 500ms |
| 第3阶段 | 50% | 30分钟 | 错误率 < 0.1%，P99延迟 < 500ms |
| 第4阶段 | 100% | 持续观察 | 错误率 < 0.1%，P99延迟 < 500ms |

每个阶段如出现异常指标，立即暂停灰度并评估是否需要回滚。

## 五、回滚机制

### 5.1 快速回滚

- 回滚方式：GitLab CI一键回滚至上一版本Docker镜像
- **回滚时间目标：15分钟内完成**
- 回滚后自动触发告警通知技术群
- 回滚后30分钟内完成根因初步分析并通报

### 5.2 数据库回滚

- 所有数据库变更必须提供回滚SQL脚本
- 回滚脚本在MR中一并提交，经DBA审查
- 数据库回滚需DBA手动执行，评估数据一致性影响

## 六、API版本管理

依据《技术架构白皮书》（文档编号：TECH-2026-005）中的API管理规范：

| 规则 | 说明 |
|------|------|
| 版本号格式 | /api/v{major}/ 路径方式 |
| 新版本发布 | 需同时维护旧版本至少6个月 |
| 废弃通知 | 提前90天在API响应头中标注Deprecation警告 |
| 强制下线 | 需通知所有接入方并确认迁移完成 |

API版本管理详情请参考《技术架构白皮书》（文档编号：TECH-2026-005）中的"API生命周期管理"章节。

## 七、发布故障应急处理

当发布导致线上故障时，按以下流程处置：

| 故障级别 | 响应时间 | 处置方案 | 关联制度 |
|----------|----------|----------|----------|
| P1（核心业务不可用） | 5分钟内响应 | 立即回滚，启动故障应急流程 | 《故障应急处理流程》（文档编号：OPS-2026-006） |
| P2（核心业务部分受损） | 10分钟内响应 | 评估后决定回滚或紧急修复 | 《故障应急处理流程》（文档编号：OPS-2026-006） |
| P3（非核心功能异常） | 30分钟内响应 | 记录问题，下一发布窗口修复 | 常规工单处理 |

P1/P2故障需按照《故障应急处理流程》（文档编号：OPS-2026-006）中的要求，完成故障报告和复盘会议。故障报告需在故障恢复后24小时内提交，复盘会议在48小时内召开。

### 7.1 发布冻结期

以下期间禁止常规发布（紧急修复除外）：
- 每月最后一个工作日（月末结算日）
- 大促活动前3天至活动结束
- 法定节假日前1个工作日
