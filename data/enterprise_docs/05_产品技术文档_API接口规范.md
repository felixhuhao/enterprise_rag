## 第一章 概述

### 1.1 文档目的

本规范定义了星辰科技所有 API 接口的统一设计标准、安全要求及运维规范，确保各产品线接口风格一致、安全可控、运维高效。

### 1.2 API 版本

- 当前版本：**v2.3.1**
- 版本命名规则：`v{主版本}.{次版本}.{修订号}`
- 主版本变更须提前 6 个月通知所有接入方。

### 1.3 关联文档

- 《信息安全策略》（03_信息安全策略）— 数据分级与传输加密要求
- 《供应商管理制度》（08_供应商管理制度）— 第三方 API 接入安全要求

## 第二章 认证与授权

### 2.1 认证方式

采用 **OAuth 2.0 + JWT Token** 双层认证机制。

| 参数 | 说明 |
|------|------|
| 认证协议 | OAuth 2.0（Authorization Code 模式） |
| Token 类型 | JWT（JSON Web Token） |
| Token 有效期 | **2小时** |
| Refresh Token 有效期 | 7天 |
| 签名算法 | RS256（非对称加密） |

### 2.2 Token 使用方式

```
Authorization: Bearer <jwt_token>
```

- Token 过期后须使用 Refresh Token 获取新的 Access Token。
- 连续 3 次 Token 刷新失败，须重新走 OAuth 授权流程。

## 第三章 接口限流策略

| 用户类型 | 限流标准 | 超限处理 |
|---------|---------|---------|
| 普通用户 | **100次/分钟** | 返回 42901 错误码 |
| 企业用户 | **1,000次/分钟** | 返回 42901 错误码 |
| 内部服务间调用 | 10,000次/分钟 | 返回 42901 错误码 |

限流基于滑动窗口算法，以 API Key + IP 为维度进行计算。

## 第四章 错误码规范

### 4.1 标准错误码

| 错误码 | HTTP状态码 | 说明 | 处理建议 |
|-------|-----------|------|---------|
| 40001 | 400 | 参数错误 | 检查请求参数格式 |
| 40101 | 401 | 未授权 / Token无效 | 重新获取Token |
| 40301 | 403 | 权限不足 | 联系管理员授权 |
| 42901 | 429 | 请求限流 | 降低调用频率，稍后重试 |
| 50001 | 500 | 服务器内部错误 | 联系技术支持 |

### 4.2 错误响应格式

```json
{
  "code": "40001",
  "message": "参数 validation 失败：field 'email' 格式不正确",
  "request_id": "req_20240315_a3f8b2",
  "timestamp": "2024-03-15T10:30:00Z"
}
```

所有错误响应须包含 `request_id`，便于问题追踪。

## 第五章 数据格式规范

### 5.1 通用要求

- 数据格式：**JSON**
- 字符编码：**UTF-8**
- 时间格式：ISO 8601（`YYYY-MM-DDTHH:mm:ssZ`）
- 金额字段：统一使用**分**为单位（整数），避免浮点精度问题。
- 分页参数：`page`（从 1 开始）、`page_size`（默认 20，最大 100）。

### 5.2 响应结构

```json
{
  "code": "0",
  "message": "success",
  "data": { },
  "pagination": {
    "page": 1,
    "page_size": 20,
    "total": 256
  }
}
```

## 第六章 文件上传

| 参数 | 规范 |
|------|------|
| 最大文件大小 | **100MB** |
| 支持格式 | PDF、DOCX、XLSX、PNG、JPG |
| 上传方式 | multipart/form-data |
| 存储位置 | OSS（阿里云对象存储） |
| 文件命名 | `{timestamp}_{random_str}.{ext}` |

- 上传文件须通过病毒扫描（ClamAV），扫描超时时间 60 秒。
- 含有敏感数据的文件，存储时须启用服务端加密（SSE-KMS）。

## 第七章 回调通知（Webhook）

### 7.1 配置方式

- 用户可在管理后台配置 Webhook URL，支持最多 5 个回调地址。
- 回调 URL 必须为 HTTPS 协议。

### 7.2 重试策略

| 重试次数 | 间隔时间 | 说明 |
|---------|---------|------|
| 第1次 | 1分钟 | 首次重试 |
| 第2次 | 5分钟 | 二次重试 |
| 第3次 | 30分钟 | 最终重试，失败后标记为投递失败 |

- 3 次重试全部失败后，系统记录投递日志，不再主动推送。
- 投递失败的回调可通过管理后台手动触发重新推送。
- 回调响应须返回 HTTP 200，否则视为投递失败。

## 第八章 数据安全要求

依据《信息安全策略》（03_信息安全策略）第四章数据分级标准：

1. **机密及以上级别**数据的传输**必须使用 HTTPS**（TLS 1.2 或更高版本）。
2. 接口返回的敏感字段（手机号、身份证号）须做脱敏处理。
3. 数据库中存储的敏感信息须使用 AES-256 加密。
4. 日志中禁止记录 Token、密码、完整卡号等敏感信息。
5. API 访问日志保留 **180天**，满足审计合规要求。

## 第九章 接口版本管理

| 操作 | 通知周期 | 说明 |
|------|---------|------|
| 新增接口 | 即时 | 无破坏性变更 |
| 废弃接口 | 提前90天 | 标记为 deprecated |
| 删除接口 | 提前180天 | 仅在主版本升级时 |
| 参数变更 | 提前60天 | 新增可选参数不视为破坏性变更 |
